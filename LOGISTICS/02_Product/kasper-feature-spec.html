Must

Tier-aware data display

Tier 1: GPS KPIs only. Tier 2: adds fuel + RPM + coolant. Tier 3: adds gauges, fault codes, AdBlue. Lower tier features greyed with "Upgrade" CTA.

Check device.tier on load. Conditionally render CAN panels. Locked features: opacity:0.4 + overlay with upgrade button. Upgrade CTA → billing flow.
Must

Live CAN gauges

RPM, coolant temp, oil pressure as live values updating every 30 seconds. Visual indicator for normal/warning/critical ranges.

Read from platform custom attributes parsed from ALLCAN300 Codec. Colour thresholds: green/amber/red configurable per parameter type in DB.
Must

Fault code translation

Raw DTC codes (SPN 3216) translated to plain English. "AdBlue level low" not "SPN 3216 FMI 18". Vendors cannot read raw codes.

DTC lookup table in DB. Key: SPN+FMI → plain text description. Source: SAE J1939 standard table. Seed with most common construction equipment codes first.
Must

Data freshness indicator

"Last data 14 seconds ago" shown always. If >5 min stale → yellow warning. If >30 min → red offline badge. Never show old data as live.

Compute (now - lastUpdate). Display relative time. Auto-refresh every 30s. If stale threshold hit → trigger offline alert.
Should

Engine immobilisation toggle

Remote engine cut-off via FMC130 digital output. Only available to account owner. Requires confirmation modal.

POST to platform /commands with type=outputControl. FMC130 DOUT1 controls relay. Two-step confirm: "Are you sure?" before sending.
Should

Share live location button

One click generates a client tracking link for this vehicle's current trip. Shareable via WhatsApp.

Generate token-based URL: /track/{token}. Store token + deviceId + expiry in tracking_links table. WhatsApp deep link: wa.me/?text=URL.

Must

Route polyline on map

Full route drawn from position history. Green start, red end, amber dot at each stop >3 min. Speed colour on line segments where overspeed occurred.

GET /reports/trips from platform for trip segments. Leaflet polyline with color array mapped from speed vs limit. Stop detection: speed=0 AND ignition=on for >3min.

Playback scrubber

Drag scrubber → vehicle marker moves along the route. Play/pause button. Speed: 1x, 5x, 10x.

Event markers on timeline

Overspeed events, geofence crossings, ignition events plotted as icons below the playback bar.

GET /reports/events filtered by trip time window. Map event types to emoji icons. Click event icon → jump playback to that timestamp.
Should

Route line changes colour: green = normal, amber = approaching limit, red = over limit. At a glance shows where speeding happened.

Split polyline into segments per position. Assign color based on speed vs speedLimit from device profile. Render as multiple Leaflet polylines.

Idle cost in AED not litres

Convert idle fuel consumption to dirhams. Formula: idle_litres × diesel_price_per_litre. Price configurable per account.

Store diesel_price in account_settings (default AED 3.19/L). Idle detection: speed=0, ignition=on, duration>configurable threshold (default 10min). Multiply idle_duration × avg_idle_consumption_rate.
Must

Fuel theft detection

Sudden fuel drop >X litres while ignition OFF = theft signature. Immediate WhatsApp alert. Configurable threshold per vehicle.

Compare consecutive fuel_level readings. If delta > theft_threshold AND ignition=off → create alert type=FUEL_THEFT. Default threshold 15L drop in <15min. Send via WhatsApp Business API.
Must

Per-vehicle consumption chart

Bar chart ranking all vehicles by fuel consumed this week/month. Outliers colour-coded amber/red automatically.

Aggregate fuel data per deviceId per date range. Sort descending. Apply colour thresholds: top 20% = red, next 20% = amber, rest = brand. Use recharts BarChart or Chart.js.
Should

Weekly WhatsApp summary

Cron job every Sunday 08:00 UAE time. Compute weekly aggregates. Format WhatsApp message template. Send via WhatsApp Business API to account owner's number.
Should

Refuel event detection

Significant fuel increase = refuel event. Log time, location, amount added. Shows vendor actual refuel history.

If fuel_level increases >10L → log as REFUEL event with timestamp and location. Store in fuel_events table. Display in vehicle timeline.

Must

Service rules by engine hours AND km

Two rule types per vehicle. Construction equipment: every 500h. Trucks: every 10,000km. Both tracked simultaneously from CAN data.

maintenance_rules table: deviceId, rule_type (HOURS/KM), interval, last_service_hours, last_service_km. Compare current_hours/current_km from live telemetry against (last_service + interval). Status = OVERDUE/DUE_SOON/OK.
Must

Red/amber/green triage list

Sorted: overdue first (red), then due-soon (amber), then healthy (green). One glance = what to book this week.

Sort maintenance_status records by priority: OVERDUE → DUE_SOON → OK. Apply left-border colour in CSS. Recalculate status on each telemetry update.
Must

WhatsApp maintenance reminders

Auto-alert at DUE_SOON and OVERDUE thresholds. Sent to vendor's WhatsApp. Optional: copy to their workshop number.

Scheduled check every 6h. On status change to DUE_SOON → send WhatsApp. On OVERDUE → send again. Store last_alert_sent to avoid repeat spamming within 24h window.
Must

Fault code → maintenance action

Active fault codes linked to maintenance action. SPN 3216 (AdBlue low) → "Refill AdBlue before next trip." Auto-generated action from fault code.

fault_to_action table: SPN → action_text + urgency. When fault detected in telemetry → lookup action → create maintenance_item. Display in maintenance screen under the vehicle.
Should

Service history log

Vendor marks service as completed → log entry with date, hours, what was done. History of all services per vehicle.

service_history table: deviceId, completed_at, hours_at_service, km_at_service, notes. "Mark complete" button resets last_service_hours/km. Show history timeline per vehicle.

Must

Alert types — all tiers

Overspeed · Geofence entry/exit (outside hours) · Device offline · Ignition on/off (out of hours) · Towing detection.

alerts table: id, deviceId, type, severity, message, lat, lng, created_at, whatsapp_sent. Computed server-side on each telemetry update. Index on deviceId + created_at for fast feed queries.
Must

WhatsApp routing rules

Not every alert needs a WhatsApp. Fuel theft + device offline = instant. Overspeed = daily digest. Configurable per account per alert type.

alert_rules table: accountId, alert_type, whatsapp_mode (INSTANT/DIGEST/NONE). Cron for digest: 18:00 UAE daily. Group alerts by type, compile message, send once.
Must

Geofence time-window logic

Geofence exit at 09:00 ≠ same severity as 23:00. Each geofence has working-hours window. Exit outside hours = alert. Inside hours = log only.

geofences table includes work_start, work_end, work_days fields. On geofence_exit event: check if current time outside window. If outside → create alert. If inside → create event log only.
Must

Auto-resolve pairing

Offline → Back online shown as a closed loop. Geofence exit → Geofence entry pairs up. Resolved alerts visually distinct from open ones.

On resolution event, update alerts.resolved_at and alerts.resolved_by_alert_id. UI: closed alerts greyed with checkmark, sorted below open ones.
Should

Alert settings panel

Vendor configures per vehicle: speed limit, geofence sensitivity, idle threshold, working hours. Kasper sets defaults at install.

alert_settings per deviceId. Fields: speed_limit_kmh, idle_threshold_min, fuel_theft_threshold_L. Default values seeded at device onboarding. UI: settings gear icon per vehicle.

6 reports, tier-gated

Report generation server-side. Input: deviceId[], dateFrom, dateTo. Output: JSON → render to PDF via pdfmake or puppeteer. XLSX via SheetJS. Tier check at API level — 403 if tier insufficient.

Utilisation = billing proof

Engine hours per asset for the billing period. For equipment rental vendors, this is the invoice evidence. Dispute-proof because data is from the device, not a driver's logbook.

Aggregate engine_hours from telemetry where ignition=on. Group by deviceId, date. Output: asset name, total hours, daily breakdown. This is the most critical Tier 3 report.

Must

No login — token URL only

Client receives a URL. Opens instantly on any phone, any browser, no app required. Link expires when trip completes or after 48h for long-hire equipment.

tracking_links table: token (UUID), deviceId, expires_at, created_by. GET /track/{token} → validate token, fetch live position, render page. No auth middleware on this route. Expiry: auto-expire on trip_end event OR time-based TTL.
Must

ETA front and centre

Calculated from current position + speed + distance to destination. Updates every 30 seconds. The client's only real question.

ETA = distance_to_destination / avg_speed_last_5min. Use Maps API (Google/Mapbox) for road distance. If vehicle stopped >10min → show "Vehicle stopped" not a fake ETA.
Must

Scoped data — no vendor info leaked

Client sees: one vehicle, current trip only, ETA, vehicle plate. They do NOT see: fuel, driver score, other fleet vehicles, history.

Tracking link API returns only: current_lat, current_lng, speed, estimated_arrival, vehicle_display_name. No other fields. Separate read-only API endpoint, never shares fleet data.
Should

Extended data for rental clients

For long-term equipment hire, vendor can optionally share: daily engine hours, idle time, utilisation summary. Vendor toggles this per client link.

tracking_links.share_extended boolean. If true: also return engine_hours_today, idle_minutes_today from telemetry. Vendor sets this when generating the link: "Share utilisation data? [Yes/No]"
Should

"Tracked by KasperGPS" footer

Every shared link is a marketing channel. The client who tracks a delivery today is a potential GPS lead next quarter.

Static footer on tracking page: "Powered by KasperGPS · kasperpro.com". Clickable → marketing landing page. Track click events for lead attribution.

Must

Driver shift start/end from ignition

Shift start = first ignition ON of the day. Shift end = last ignition OFF. If driver switches vehicles mid-day, system creates separate rows per vehicle segment.

Query positions table: group by driver_id + device_id per calendar day. First ignition_on = shift_start. Last ignition_off = shift_end. If same driver appears on multiple devices → multiple rows, sum in totals. Driver assignment: drivers table linked to device via shift_assignments (driver_id, device_id, date).
Must

Moving / Idle / Stopped breakdown

Per driver per day: total hours split into moving (speed>5), idling (speed=0, ignition ON), stopped (ignition OFF). Shown as mini bar chart in the row.

Iterate positions for driver's shift window. Classify each position record: MOVING (speed>5), IDLE (speed≤5 AND ignition=on), STOPPED (ignition=off). Sum duration between consecutive pings. Store daily_summary: moving_mins, idle_mins, stopped_mins per driver_id per date.
Must

Date range selector

View one day, one week, or custom date range. Weekly view shows a row per driver per day with totals. Monthly aggregates for payroll.

Date picker: single day, week preset, custom range. Query daily_summary table with date BETWEEN. Group by driver_id for multi-day. Sum moving/idle/stopped across date range. Limit: 31 days max per query to avoid timeout.
Must

Export to Excel / PDF

Download timesheet for the selected date range. Used for payroll processing and client billing proof for equipment hire.

Server-side: generate XLSX via SheetJS. Columns: driver, vehicle, date, shift_start, shift_end, total_hours, moving_hours, idle_hours, stopped_hours. PDF via pdfmake with Kasper header. Endpoint: GET /timesheet/export?from=&to=&format=xlsx|pdf
Must

Driver assignment to vehicle

Vendor assigns a driver name to each device. Driver can be reassigned when they switch vehicles. Timesheet tracks by driver name, not just vehicle.

drivers table: id, account_id, name, phone. shift_assignments: driver_id, device_id, assigned_from, assigned_to (nullable = current). UI: per device → "Assigned driver" dropdown. Change triggers new shift_assignment row with timestamp.
Should

Overtime detection

Flag shifts exceeding a configured threshold (e.g. >10 hours). Highlighted in amber. Vendor can set per-driver or fleet-wide threshold.

account_settings.overtime_threshold_hours (default 10). On daily_summary compute: if total_hours > threshold → flag overtime=true. Show amber highlight on row. Include in weekly WhatsApp summary: "2 drivers exceeded 10h today."
Should

WhatsApp daily summary to vendor

End-of-day WhatsApp: "Today: 5 drivers, 44.9 total hours, 7.3h idle. Longest shift: Suresh P — 9.7h." Sent at 20:00 UAE.

Cron 20:00 UAE daily. Aggregate all driver shifts for the day. Format WhatsApp template message. Send to account_owner.phone. If any driver overtime flagged → include in message: "⚠ Overtime: [driver name]".
Could

Idle time cost per driver

If Tier 2/3: translate each driver's idle hours to AED wasted. "Ramesh: 3.8h idle = AED 142 in fuel." Connects idling to individual accountability.

Join driver_daily_summary with fuel dashboard idle cost calculation. idle_cost = idle_hours × avg_idle_fuel_rate × diesel_price. Only available for devices with LVCAN200/ALLCAN300. Display as additional column (tier-gated).
